Build Module 13: Evidence & Readiness Assessment for the existing EU AI Act Compliance Readiness Dashboard.

Flowchart section: none (practical add-on, not in the Future of Life flowchart). Legal basis: Article 9-15, 17, 26, 43, 47-49, 72-73; Annex IV (technical documentation), Annex VIII (registration). Depends on: Module 12. Feeds: Module 14, Module 15.

Apply the shared conventions from Module 4. This module does NOT re-derive whether an obligation applies (that is Module 12) — it only assesses readiness STATE (evidence in place / partial / gap / not started / unknown) per applicable obligation and rolls the rows into a system-level readiness score. It reuses Module 12's obligation set, role-conditional obligations, registration and conformity flags, and carries through each row's legal_basis_citation, applicable_from_date and guidance_status.

Routes: /readiness (list, one row per system × obligation) and /readiness/systems/[id] (detail — per-system evidence checklist grouped by obligation; each row shows required artifacts, answered evidence state, the gap, legal basis + deadline badge, and a "create remediation item" hand-off to Module 14).

Questionnaire: one evidence question per applicable Module 12 obligation row, seeded from Module 12; only shown for obligations that actually apply. Options for every question: In place · Partial · Not started · Not sure · Not applicable.
- OBL-ART9-RMS: Do you have a documented risk-management process, kept up to date across the lifecycle? Required: risk-management plan/policy, maintained risk register, residual-risk analysis, testing records.
- OBL-ART10-DATA: Do you have data governance documentation for training/validation/testing data? Required: data governance policy, dataset documentation, provenance records, bias/representativeness examination.
- OBL-ART11-TECHDOC: Does an Annex IV technical documentation file exist and is it current? Required: the Annex IV file (must exist before market placement, maintained).
- OBL-ART12-LOGGING: Does the system automatically log events, with logs retained under a retention policy? Required: logging design/spec, retained logs, retention policy.
- OBL-ART13-TRANSP-DEPLOYER: Have you prepared instructions for use / transparency documentation for deployers? Required: deployer-facing instructions (capabilities, limitations, human oversight, declared accuracy).
- OBL-ART14-OVERSIGHT / OBL-ART26-DEPLOYER: Are human-oversight measures designed in and documented (and, if deployer, operationally assigned)? Required: oversight design, operator instructions, assigned oversight roles/training.
- OBL-ART15-ACCURACY: Do you have accuracy, robustness and cybersecurity evidence? Required: declared accuracy metrics (stated in instructions for use), robustness/adversarial test reports, cybersecurity measures/threat model.
- OBL-ART17-QMS: Is there a documented QMS covering this system? Required: QMS documentation, procedures, responsibilities, record-keeping.
- OBL-ART43-CONFORMITY: Has conformity assessment been completed with an EU declaration of conformity and CE marking (plus notified-body certificate where required)? Required: conformity assessment records, signed EU declaration of conformity, CE marking evidence, notified-body certificate (Annex VII route only).
- OBL-ART49-REGISTER / OBL-ART6-3-NOTHR-DOC: Is the system (or the Art 6(3) not-high-risk self-assessment) registered in the EU database? Required: registration entry (Annex VIII); for a not-high-risk derogation, the documented Art 6(3) self-assessment must also be registered; note the non-public variant for law-enforcement/migration/border.
- OBL-ART72-PMM: Do you have a post-market monitoring plan and are you collecting monitoring data? Required: documented plan + evidence of ongoing data collection (recurring control, not a one-off).
- OBL-ART73-INCIDENT: Is there a serious-incident reporting procedure with an owner and defined timelines? Required: documented procedure, assigned owner, awareness of statutory deadlines (readiness = procedure + owner).

Deterministic rules (src/lib/evidence/evidenceRules.ts): for each applicable Module 12 obligation, compare a static required-artifact list to the answered evidence state to derive a per-obligation readiness_status. No LLM reasoning.
- readiness_mapping: In place → evidence_in_place; Partial → partial_evidence; Not started → evidence_gap; Not sure → unknown_evidence_state; Not applicable → not_applicable.
- gap_scoring_logic: system readiness_score = 100 × (weighted evidence_in_place obligations) / (applicable obligations), where partial counts 0.5 and unknown counts 0. Obligations with a nearer applicable_from_date are weighted higher for prioritisation (not for the raw score).
- Carry through legal_basis_citation, applicable_from_date and guidance_status per obligation from Module 12 (high-risk rows = provisional with 2027-12-02 / 2028-08-02 dates; Art 6(3) row = draft) so Module 14/15 can cite the source and see which deadline drives urgency. source_version_date 2026-07-07.

Result object:
- classification_or_status: per-obligation readiness label (evidence in place / partial evidence / evidence gap / not started / unknown / not applicable) plus a system-level "X of Y applicable obligations have evidence in place" summary. Factual readiness wording only — never "compliant".
- confidence_score (measures confidence in the readiness picture, not compliance): start 100; −15 per obligation answered 'Not sure', −20 if the parent Module 12 obligation set is itself 'needs review', −10 if required artifacts for an applicable obligation were left unanswered. Standard labels.
- reasoning_summary (e.g. "Of 12 applicable obligations, 4 in place, 3 partial, 5 gaps; nearest provisional deadline 2027-12-02 drives priority; technical documentation and conformity assessment are the largest gaps"), positive/negative indicators, key_uncertainties (deadlines are provisional; conformity route self-assess vs notified body; non-public registration variant), missing_fields, recommended_next_questions.
- role_conditional_obligation: INHERITED from Module 12 — the checklist lists only obligations for the roles the system holds (a provider checklist is far longer than a distributor one). Re-runs if a Module 9 reclassification changes the role set.
- reclassification_trigger_flags / registration_required / standards_conformity_route / not_high_risk_documentation_flag: all inherited from Module 12 and drive specific evidence rows (e.g. registration entry exists yes/no; notified-body certificate required; Art 6(3) self-assessment + registration).

UI: summary cards (total applicable obligations, evidence in place, partial, gaps, not started, unknown, obligations past/near a deadline, overall readiness %); table one row per system × obligation (system, obligation, legal basis, applicable from, guidance status, required evidence, readiness status, confidence, key uncertainty, owner if set, actions); CSV exports the per-obligation evidence rows with required_artifacts, readiness_status, legal_basis_citation, applicable_from_date so Module 14/15 consume them directly; filters (readiness status, obligation, role, guidance status, applicable-from range, confidence, owner assigned). Cross-module consistency warning: flag when the parent Module 12 set is 'needs review' or an upstream classification is uncertain (gaps may be over/under-stated until confirmed).

Implementation: rules in src/lib/evidence/evidenceRules.ts; helpers buildEvidenceChecklist(system) (from Module 12 obligation set), scoreReadiness(system), buildReadinessAssessment(system). Do NOT recompute applicability — read Module 12. Self-check tests: high-risk system with technical documentation missing → OBL-ART11-TECHDOC evidence_gap, readiness_score reduced; all applicable obligations 'In place' → readiness ~100; obligation answered 'Not sure' → unknown_evidence_state, confidence reduced; Art 6(3) derogation → checklist includes the self-assessment + registration evidence row.

Acceptance criteria: /readiness list + detail; reads Module 12 obligation set without recomputing applicability; one evidence question per applicable obligation; deterministic readiness_status + system readiness score; inherited citations/dates/guidance_status carried through; disclaimer footer; CSV export consumable by Module 14; consistency warning on 'needs review' parent matrix; tests cover the readiness mapping and scoring.
